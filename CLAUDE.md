## What this is

**TWIST** (Tracking via World-model State Transitions) reframes dense surgical tissue-point tracking as a **passive, observation-corrected state-space world model**: tracked points are the *state*, the next frame is the *exogenous input*. A separable learned **transition prior** (frame-free tissue dynamics, handles occlusion by rollout) is corrected by an **observation update** from each new frame. (A `CORE.md` concept doc is referenced historically but was never checked in â€” the concept/novelty summary above plus the "v2 improvements" section below are the current source of truth.)

**Hardware availability.** Up to **8Ã— A100 80GB** GPUs are available, each scheduled as a separate single-GPU SLURM job (no multi-GPU/DDP runs â€” one run per GPU). This is what makes the `jsub -w` sweep-agent pattern the workhorse: submit the same agent job up to 8Ã— to run 8 cells (or 8 independent runs) fully in parallel. The historical "1Ã— A100 40GB" budget is superseded; the A100-40 sizing notes (e.g. BATCH_SIZE 2â€“3 @ 512) still apply on the rare occasion a 40GB card is used, but default to 80GB sizing (BATCH_SIZE 6 @ 512, ~75 GB peak).

## Architecture

**Single orchestrator, two entrypoints.** Everything routes through `main.run_pipeline(mode, config)`. `train.py` calls it with CLI parsing; `sweep_agent.py` calls it with `config=dict(wandb.config)` (the *exact same path*). `run_pipeline` loops over schedule **stages**: per stage it builds datasets â†’ dataloaders â†’ model + loss â†’ runs `Engine(...).fit()` â†’ checkpoints â†’ marks the stage complete.

**Config system** (`utilities/config.py`). Configs are **W&B-sweep-format YAML** (`parameters: {KEY: {value: ...}}`), flattened to `{KEY: value}` and loaded into a `DotMap`. The same file therefore drives a plain run and a sweep. CLI `--KEY=val` and dotted `--A.B.C=val` overrides are coerced to the existing value's type; W&B sweeps deliver nested overrides as flat dotted keys that get folded back into the nested structure. `boot_mode` (`-b`) shrinks batch/epochs/datasets and forces the no-download CNN encoder (login node has no GPU/network for DINOv3).

**Phased training schedule.** A config's `STAGES` list is an ordered schedule (Kubric pretrain â†’ long-horizon â†’ surgical â†’ â€¦). `resolve_stage_config(cfg, i)` overlays stage `i`'s override block onto the base config â€” **top-level keys replace, not merge** (so a stage's `DATASETS` and `MODEL` fully define that phase). A config without `STAGES` is one implicit stage. Run state lives in `$RESULTS_DIR/<EXPERIMENT_NAME>/run_state.json` (`utilities/runs.py`); `--resume-run` continues at `first_incomplete_stage`. Cross-stage: a rerun of the same stage resumes from `last.pt` (model+optim+scaler+epoch); a fresh later stage **carries the previous stage's weights** (`strict=False`).

**Datasets** (`dataset/`). `DATASET_DEFAULTS` in `dataset/wrappers.py` is the single source of truth for *where* each dataset lives (`ROOT_DIR`) and *which reader* serves it (`READER`); configs only override per-experiment sampling. `create_datasets_from_config` merges defaults < `ALL_DATASETS` overrides < per-dataset config, splits sequences (fractional+seed or explicit lists), and forwards only the config keys the reader's `__init__` accepts (UPPER_CASE â†’ snake_case). **`VAL_SUBSAMPLE`** (top-level global + per-dataset override; 2026-07-13) keeps only a representative fraction of EACH dataset's *val* clips in the per-epoch `validate()` set â€” `None`/`>=1` = all, `0<f<1` = evenly-spread deterministic fraction, `<=0` = drop from per-epoch val (still scored by benchmark eval). It is **decoupled from `VAL_FRACTION`** (the train/val split ratio) and applied to the val reader only; it exists because eval benchmarks set `VAL_FRACTION: 1.0` had made per-epoch val bigger than train (~10k clips, ~90 min/epoch of silent validation that looked like a hang). Impl = `subsample` reader kwarg (`dataset/base.py` + `cotracker.py::_build_index`); verified in `workshops/37`. Every reader yields the **canonical tracking item dict** documented in `dataset/__init__.py`: `frames (T,3,H,W)`, `tracks (T,N,2)` pixel xy, `visibility (T,N)`, `pos_valid (T,N)` (frames whose *coords* are supervisable: visible, or occluded-but-in-frame on `HAS_OCCLUDED_GT` datasets â€” Kubric stores real coords on its ~28% occluded frames), `queries (N,3)=(t,x,y)`, `frame_size (2,)`, `video`, `clip_idx`, optional `depths`. `RESIZE_MODE` selects the geometry: `"cover"` (default; aspect-preserving resize + center-crop, the internal harness) or `"stretch"` (anisotropic squash, NO crop â€” the **canonical TAP-Vid protocol**; use it at 256Â² for numbers comparable to published tables; the cover crop discards ~44% of DAVIS width and cost CoTracker3 Î´ 0.849â†’0.58 in-harness). **Every dataset (including CT3Kubric) is now served by the single shared reader** `dataset/cotracker.py::CoTrackerTracksDataset` over the common `index.json` + per-clip `.npz` layout â€” there is no longer a dataset-specific reader. CT3Kubric ships in a bespoke per-sequence layout, so `ct3kubric_data_prep.py` **converts** it into the shared layout (transpose to frame-major, embed frames, drop the model-unused depth) under `DATA/CT3Kubric/cotracker_tracks`; the other `*_data_prep.py` scripts produce the same layout directly. **IO fast-path (2026-07-13):** the per-clip `.npz` is DEFLATE-compressed, so `np.load` must decompress the *whole* frames/tracks member (~1.8 s/clip on Kubric) even to read a 24-frame window â€” which starved the GPU on the 8â€“12-CPU single-GPU jobs. `assets/dataprep/transcode_to_mmap.py` rewrites each clip's arrays as **uncompressed per-array `.npy`** in a sibling `<clip>/` dir (`frames.npy`/`tracks.npy`/`visibility.npy`/`queries.npy`; originals kept, ~2Ã— disk); the reader auto-prefers that dir and reads only `frame_ids` via `np.load(mmap_mode='r')` (~0.02 s/clip, **~80â€“130Ã— faster**), falling back to `.npz` when absent â€” a pure, reversible, config-free speedup. `dataset/base.py::_apply_resize` also **skips the identity `F.interpolate`** when the stored size already equals `target_size` (Kubric 512Â²â†’512Â²). Run the transcode on each `ROOT_DIR` (login node has 256 cores): `python assets/dataprep/transcode_to_mmap.py --root <ROOT_DIR> --workers 64` (verified in `workshops/35`).

**World model** (`models/`, learned predictorâ€“corrector state space). `TrackerWorldModel`: frozen encoder â†’ frame-free `TransitionModel` (per-point GRU + inter-point self-attention, bounded tanh displacement; `TRANSITION.VIS_INPUT` adds the visibility belief to the token) â†’ `ObservationModel` (cost-volume soft-argmax + cross-attention heads) â†’ per-point Kalman-gated fusion. **v2 observation flags** (default off = v1 architecture, checkpoint-compatible): `OBSERVATION.STATS` feeds detached correlation peak/entropy/margin + the prior's log-scale to the vis/gate/logvar heads; `OBSERVATION.COARSE` adds the **global re-acquisition stage** (template vs the full feature grid, confidence-gated window re-centering â€” snap-back beyond the Â±24px window; zero-init gate bias âˆ’2 starts prior-centered). `forward` supports per-point `observe_mask (T,N)` (simulated occlusion: masked points ride the prior while neighbours observe) and exports `observed`/`corr_valid`/`corr_logits`/`win_center` (+ `gcorr_logits` train-only) for the loss. `models/encoder.py` `FrozenFrameEncoder` wraps DINOv3 (HF-cached/offline; preprocessing is now **on-device** â€” no PIL round-trip) with a no-download CNN fallback for CPU/smoke/boot. `models/losses.py` `TrackerLoss` = **direct Huber position (must dominate; `HUBER_DELTA` 0.02 â‰ˆ 5px @512 = L1-like beyond that)** on coords+prior+rollout, all masked with `pos_valid` (**through-occlusion supervision**; `USE_OCCLUDED_GT` toggles) + visibility BCE (observed point-steps only) + **correlation CE** (`CE_WEIGHT` local / `GLOBAL_CE_WEIGHT` global â€” matcher gradient that bypasses the gate) + **decoupled uncertainty NLL** (`UNC_WEIGHT`; trains post+prior logvars against detached errors) + KL(postâ€–prior) (**retired: pins at the free-bits floor â€” run `KL_WEIGHT: 0`**; `kl_raw` logs the pre-clamp value). `models/metrics.py` `tracking_metrics` = TAP metrics (EPE, Î´_avg, OA, AJ); headline is `val/epe`. Models are built via `create_model_from_config`/`create_loss_from_config` (dynamic `getattr(models, MODEL_CLASS)`), never instantiated directly.

**Training engine** (`utilities/engine.py`). Trains one stage end to end. Per-component optimizer groups (`encoder.*` â†’ `RGB_ENCODER_LR`; rest â†’ `LR`; `RGB_ENCODER_LR==0` or `FREEZE_BACKBONE` â†’ encoder frozen, no group). AMP = bf16 on A100 / fp16+GradScaler elsewhere / off on CPU. All schedules are **pure functions of epoch** (resume-safe): cosine LR + linear warmup, KL weight annealed `KL_WEIGHT_START`â†’`LOSS.KL_WEIGHT`, teacher forcing for the first `TEACHER_FORCING_EPOCHS`, observation dropout ramped over `OBS_DROPOUT_EPOCHS`. **Observation dropout is per-point and occlusion-shaped**: contiguous spans of `OBS_DROPOUT_SPAN=[3,8]` frames per point (expected dropped fraction = `OBS_DROPOUT`), so masked points coast on the transition while neighbours keep observing â€” the inference-time occlusion regime. The rollout loss's observe length is fixed (`ROLLOUT_OBSERVE_STEPS`) or resampled per batch from `ROLLOUT_OBSERVE_MIN..MAX`; rollout *eval* always uses the fixed protocol. Diagnostics: `gate_vis` vs `gate_occ` (mean Kalman gate split by GT visibility â€” the gate-health chart; a working gate is high on visible, â†’0 on occluded), `epe_occ` (EPE on occluded-but-supervised frames), `coarse_gate`, `kl_raw`. Per epoch: train â†’ validate â†’ checkpoint `last.pt`/`best.pt` under `$RESULTS_DIR/<run>/stage{idx}_{name}/`. W&B opens one run per schedule and logs per-epoch scalars + a periodic pred-vs-GT `wandb.Video` (`utilities.visualization.render_comparison_frames`). **Hot-loop perf (2026-07-13):** the per-step compute timing (`performance/ms_per_image` etc.) uses **CUDA events** read once at epoch end rather than two per-step `torch.cuda.synchronize()` (which were serializing CPU/GPU overlap); the grad norm is taken from the value `clip_grad_norm_` already returns (`_grad_norm_and_clip`) instead of a redundant per-parameter reduction. Both preserve logged values (grad-norm parity verified in `workshops/35`).

### v2 improvements (2026-07 diagnosis â€” branch `twist-v2-improvements`)

The 2026-07-02 diagnosis found the model was never *trained* on its thesis skill: the loss masked all position terms to visible frames (discarding Kubric's ~28% occluded-but-in-frame GT), dropout blinded whole frames (never the some-points-blind regime real occlusion creates), the gate had no gradient at occluded frames, re-acquisition was structurally capped at the Â±24px window, `HUBER_DELTA 0.2` (~51px) gave vanishing tight-threshold gradients, and the KL/logvar machinery was inert. The v2 changes (each verified in `workshops/26â€“28`): **pos_valid through-occlusion supervision** (F1), **per-point occlusion-shaped dropout** (F2), **coarse re-acquisition stage** (F3), **HUBER_DELTA 0.02 + correlation CE** (F4), **functional uncertainty + stats-fed gate, KL retired** (F5), on-device encoder preprocessing, canonical TAP-Vid `stretch` mode, STIR MEAN exclusion. **UPDATE 2026-07:** the all-on `train_v2` run (xrj5v46a) **collapsed** â€” HUBER_DELTA 0.02 + UNC_WEIGHT 0.5 broke the "direct position Huber must dominate" invariant (w_unc â‰ˆ âˆ’45 vs w_pos â‰ˆ 0.02, negative total loss, gate pinned). v2 features are therefore re-earned **one at a time** on the stable base `config/train_best.yaml` (= snowy-sweep-1 `fqbpeml7`: RAD24/MC0/RW2, HUBER 0.2, UNC 0): `config/sweep_supervision.yaml` (2Ã—2 occGTÃ—huber, v1 arch) + `config/ablation_{ce,coarse,redetect}.yaml` (one v2 block each, 20-ep ranking runs via `sbatch .sbatch_scripts/train_cfg_a100_80.sbatch config/<file>.yaml`), alongside the `twist_best_kubric_r512` 50-ep long run. `USE_OCCLUDED_GT` now defaults **True** in the loss. ### Eval protocol (must match CoTracker / TAP-Vid to be comparable)

We follow the **TAP-Vid "queried first"** protocol (the one CoTracker reports): each GT point is **queried at its first *visible* frame**, and metrics are scored **only on the frames strictly after that query frame** (`EVAL_QUERY_MODE: first`, the benchmark default). This matters because the readers store **occluded GT coordinates as `(0,0)`** (placeholder; `visibility=False`) â€” and ~**17 % of TAP-Vid-DAVIS** and ~**24 % of RoboTAP** points are occluded at frame 0. The TWIST model takes one query frame per forward, so `first` mode groups a clip's points by their first-visible frame and runs **one forward per group** (timing's ms/frame still uses a single all-points pass).

How occluded GT is (correctly) handled in `models.metrics.tracking_metrics`: occluded frames are **masked out by GT visibility** â€” `epe`/`delta` sum only over visible-and-evaluated points, and in Average Jaccard an occluded point can only ever be a *false positive* (if the model predicts it visible), never a distance term â€” so the `(0,0)` placeholder **does not** corrupt the metric *at occluded frames*. The damage the `(0,0)` does is only as a **query coordinate** (a point queried at frame 0 while occluded there is tracked from `(0,0)`); querying at the first *visible* frame is the fix. Definitions match TAP-Vid exactly: `OA = mean(pred_vis == gt_vis)` over evaluated frames; `Î´_avg` = fraction of visible points within {1,2,4,8,16}px; `AJ = TP/(TP+FP+FN)` combining position + visibility.

### Post-occlusion recovery (POR)

Beyond the TAP headline, eval reports **Post-Occlusion Recovery** (`models.metrics.recovery_metrics`, wired through `utilities/evaluation.py`) â€” how well the model *re-acquires* a point **after** an occlusion, the project's core failure surface. Per **occlusion event** (a maximal GT-occluded run beginning after a point's first-visible frame) it scores **snap-back** (the first re-emergence frame, W=1) and **sustained** (the contiguous visible run after it, capped at `EVAL_RECOVERY_WINDOW`, default 8), each as EPE (px â†“) and Î´ (within {1,2,4,8,16}px â†‘), aggregated **length-weighted** with `w(L)=L` (longer occlusions count proportionally more) and always strata'd by occlusion length. Two GT-availability cases: **Case A** (real data; occluded coords stored `(0,0)`) scores recovery on **visible frames only**, so the placeholder is never read; **Case B** (datasets flagged `HAS_OCCLUDED_GT: True` in `DATASET_DEFAULTS` â€” KUBRIC / DYNAMICREPLICA / POINTODYSSEY) *additionally* scores the **through-occlusion** track (`tho_epe`/`tho_delta`) and the **drift** curve (EPE vs frames-since-onset) â€” the rollout-quality measures (ties to the multi-step-rollout-loss work). It is distinct from `occlusion_accuracy` (a visibility classifier) and from Î´-on-occluded-frames (tracking *while* hidden). Reported in **its own artifacts** so `evaluation.csv` stays CoTracker-comparable: `recovery.csv` (headline POR/THO scalars), `recovery_by_length.csv` (the curve), `recovery_drift.csv` (Case B), plus a W&B table + `eval/<dataset>/por_*` scalars. Returns **sufficient statistics** pooled globally over clips (events are per-point), reduced once via `finalize_recovery`. Toggle with `EVAL_RECOVERY` (default on). Built + verified in `workshops/24_*` (metric, exact known-answer cases) and `workshops/25_*` (evaluator wiring).
